i got this reviews as weLll:
1. Onboarding atomicity vs. Google Sign-In session
You say "The user is not considered signed in until the form is submitted." But Supabase Auth will have a valid session the moment OAuth completes — that's how the backend knows who's filling the onboarding form. What you actually mean is "no users row exists until onboarding submits." Those are different things. Rewrite to: "Supabase session exists post-OAuth, but the app treats the user as unauthenticated for all routes except /onboarding until the DB user record is created."
2. "Close app mid-onboarding → restarts from scratch" contradicts Supabase session persistence
Supabase will restore the session on next launch. So on return, the user hits /onboarding again with a valid Supabase session but no DB row. That's fine — but "restarts from scratch" is misleading. The form is empty, but they don't re-do Google OAuth. Clarify.
3. /profile/me — "backend-handled redirect"
A backend redirect requires the backend to know the logged-in user from the request. With Supabase Auth + a SPA/Next.js app, the JWT is in the client. You'd need either (a) middleware that reads the Supabase cookie server-side, or (b) a client-side redirect. "Backend-handled" is ambiguous and probably wrong for a Next.js app — this is almost certainly a middleware or client-side concern. Pick one.

By the end of this setup, your project will automatically check and format your code before every commit, ensuring consistent code quality across your team.

What You’ll Be Setting Up
ESLint: A static code analysis tool that identifies problematic patterns in JavaScript/TypeScript code
Prettier: An opinionated code formatter that enforces consistent code style
Husky: Git hooks made easy — runs scripts before commits to ensure code quality
lint-staged: Runs linters and formatters only on staged files for faster execution

A basic index.ts file to get you started

- What type of modules your project uses (ESM/CommonJS)
- Which framework you’re using (if any)
- Whether you’re using TypeScript
- Where your code runs (browser/Node.js)

What is Husky?
Before diving into the setup, let’s first explore Husky and its capabilities. Husky is a Git hook tool that automates tasks before commits and pushes, ensuring code quality and consistency. Apart from running ESLint on staged files, it can:

Run Tests Before Commit — Prevents broken code by executing unit or integration tests.
Enforce Code Formatting — Runs Prettier or other formatters to maintain style consistency.
Prevent Bad Commits — Blocks commits with linting errors, security vulnerabilities, or missing dependencies.
Automate Dependency Checks — Ensures packages are installed before committing or pushing.
Enforce Commit Message Standards — Uses commitlint to maintain a structured commit history.

4.1 Configuring package.json
Understanding the prepare Script
The prepare script in package.json ensures that Husky is correctly installed and configured, but it also includes a conditional check:

{
   ...
   "prepare": "node -e \"if (process.env.NODE_ENV !== 'production'){process.exit(1)} \" || husky install | chmod ug+x .husky/*"
}
Breakdown:

Check Environment: The node -e command runs an inline Node.js script to check if NODE_ENV is set to production. If it's not, the script exits with a failure status (1).
Install Husky: The || operator ensures that if the previous command fails, husky install runs, setting up Git hooks.
Ensure Execution Permissions: The | chmod ug+x .husky/* command ensures that Husky scripts are executable.
This setup prevents Husky from running in production environments while maintaining proper permissions for pre-commit hooks.

Inside your package.json, define a script for running lint-staged:

File: .lintstagedrc.json

// Ref: https://nextjs.org/docs/pages/api-reference/config/eslint#running-lint-on-staged-files

const buildEslintCommand = (filenames) =>
